# AGENTS.md

EasyPrompt — a Next.js app for AI prompting built on a **two-object content model** (migration
`0007`): **Templates** (reusable frameworks with variables — created in the Builder or the form
editor) and **Prompts** (ready-to-use, generated by filling a Template, or imported/manual/AI). The
main public experience is browsing the Template and Prompt catalogs. Users build their own only when
the catalogs do not fit. The catalog is static + data-driven; an optional Supabase-backed account
system adds a unified **My Library** of saved Templates + Prompts with private/public visibility. The
anonymous builder works without any account (gated by `isSupabaseConfigured()`). See "Content model"
below.

## Non-negotiables
- Use product language in UI copy: **Template** and **Prompt**. Do not expose storage/internal words
  like notebook, saved, authored, or table names.
- Templates are empty constructors. Opening a Template starts blank unless the user explicitly restores
  a draft.
- Prompts are ready-to-use content. Template-sourced Prompts may link back to the source Template;
  standalone/manual/import Prompts should not show a source Template link.
- `visibility` is the source of truth for private/public state. Treat legacy `is_public` columns as
  implementation baggage.
- Do not read signed-in auth state in `app/layout.tsx`; it can make static routes dynamic.
- CSS imported in App Router is global. Prefix route CSS under a page-root class.
- Supabase migrations must be applied before matching code is considered done, and
  `lib/supabase/types.ts` must stay in sync.
- Small UI changes do not need browser verification by default. Broad layout or interaction-heavy
  changes do.

## Stack
- Next.js 15.5.19 (App Router) · React 19 · TypeScript 5.7 (strict)
- Supabase (`@supabase/ssr` + `@supabase/supabase-js`) for auth + Postgres; `zod` 4 for input validation
- Vitest 3 for tests (node environment, globals on)
- `@/*` path alias → repo root (see `tsconfig.json`; mirrored in Vitest via `vite-tsconfig-paths`)

  (`scripts/mint.ts`; `--plan pass --days 30` for a time-limited pass). Same secret as the verifier.

## Working efficiently in this repo
- **Behavior lives in `lib/` + `data/`, and it's pure.** To understand or change logic, read the
  relevant `lib/<area>/` module and its `tests/<area>.test.ts` — not the React tree. The `app/` and
  `components/` layers are thin shells over these.
- **One schema file:** `lib/supabase/types.ts` lists every table + RPC. Read it instead of opening
  all migration SQL files; reach for a specific `supabase/migrations/000N_*.sql` only when you
  need the exact DDL/policy.
- **Targeted over broad:** the file/area names are descriptive (`lib/ratings`, `lib/community`,
  `lib/metrics`…). Prefer a scoped Read/Grep to a repo-wide sweep; `node_modules`, `.next`, and
  `.agents/skills` are large and irrelevant.
- **Tests are the contract editor:** catalog/template invariants, schemas, and mappers are all
  unit-tested — change the data, run `npm run test`, let the failures guide you.
- **Database migrations are agent-applied:** when adding a Supabase migration, Codex must apply it to
  the project database before calling the task done. Before applying, verify the target project,
  confirm the migration is not already applied, inspect the SQL for destructive or unsafe behavior,
  and run a post-apply verification query.
- The three `docs/` files (`ARCHITECTURE`, `ENGINEERING`, `DESIGN-SYSTEM`) are deeper references.
  Do not reread all docs by default. Read or update the relevant doc when the task touches that area.
  Keep `AGENTS.md` focused on rules future agents must see.

## UI verification discipline
- Do not launch a dev server or browser visual check for every UI change.
- For small or medium UI changes, rely on code inspection plus targeted type/build/test checks when
  they are useful. The user will do visual confirmation.
- Use browser visual verification only for complex, broad, high-risk, or interaction-heavy UI work
  where rendered behavior is hard to reason about from code.
- If the need for visual verification is unclear, ask the user before starting a dev server.
- Do not start `npm run dev`, `npm run start`, or browser tooling solely because a UI file changed.

## Architecture (data flow)
1. `data/types.ts` — the content schema: `Template`, `Field` (text/textarea/select/pills),
   `Checkbox`, `Category`. Source of truth for the *shape* of content.
2. `data/templates.ts` — the catalog (`TEMPLATES`, `CATEGORIES`) plus pure helpers:
   `getTemplate`, `categoryLabel`, `countFor`, `relatedTemplates`, `questionCount`, `displayTitle`.
   Category counts are derived at runtime, never hard-coded.
3. `lib/buildPrompt.ts` — pure prompt assembly: `buildPrompt`, `defaultAnswers`, `blankAnswers`, `answeredCount`,
   `openInUrl`. Assembly order: base prompt → filled fields (blank fields are skipped — the "smart
   exclusion" rule) → checked boxes. `blankAnswers` is the fill-in UI starting state: every field is
   empty and every checkbox is off. `defaultAnswers` is only an authored suggestion/reference for
   tests and template validation. Token estimate is `ceil(text.length / 4)`.
4. `app/` — App Router pages. **`app/templates/`** and **`app/prompts/`** are the primary public
   browsing surfaces. Builder routes are secondary creation surfaces for users who need to make their
   own Template or Prompt. **`app/templates/`** is the Template catalog: `app/templates/[slug]/` is
   the per-template fill→generate builder, **statically generated** (`generateStaticParams`,
   `dynamicParams=false`) for SEO (`Builder.tsx`, `PromptsClient.tsx` client components).
   **`app/prompts/`** is the Prompts catalog — a real browse/copy/customize library of **curated
   example Prompts** (`data/prompts.ts`, static). It has a **distinct dark `.md`-file identity** (vs the
   light Templates picker) so the two never blur: `PromptCard` renders a dark markdown-file tile and the
   `--prompt-accent` (terminal green) / `--bg-md` tokens mark Prompts everywhere. Page root stays
   transparent (ambient grid preserved); only the cards/editor go dark. `app/prompts/[slug]/` resolves
   in order: legacy catalog slug → 308 to
   `/templates/[slug]`; **example Prompt** (SSG via `generateStaticParams` + `dynamicParams=true`) →
   `PromptDetail`; else a public user Prompt by share slug → `CommunityPrompt`. **`app/build/`** is a
   create hub: `/build` is the **Builder Overview** (anon-safe; "+ New Template" / "+ New Prompt", plus
   recent creations when signed in), `/build/template` is the block builder (the old `/build`, kept
   static + anon via `?from=`), `/build/prompt` is the **single-pane Markdown Prompt editor**
   (`PromptEditor`; anon-safe — local draft + Copy/Open-in logged out, Save prompts sign-in). The
   editor is **direct-edit with live highlighting**: a transparent `<textarea>` over a syntax-colored
   `<pre>` mirror (`lib/markdown/highlight.ts`, character-preserving so the caret never drifts in the
   monospace font) — no separate preview pane.
   Account routes (`app/(auth)`, `app/my`, `app/account`) are dynamic; `app/my` is the unified library
   (filters via `?filter=`). Opening a saved Prompt (`/my/prompts/[id]`) shows the **prompt itself**
   (manual → `PromptEditor`; template-sourced → `SavedPromptView` with a secondary "Edit answers" →
   `/my/prompts/[id]/edit` form) — never the fill-in form as the primary view. Share routes:
   `/p/[slug]` (Template/notebook), `/s/p/[slug]` (Prompt). CSS is per-route, imported by its page.
5. `components/` — presentational + client islands. `components/iconNames.ts` owns the runtime
   `ICON_NAMES` array and the derived `IconName` type (`Icon.tsx` re-exports the type).

## Content model (Templates & Prompts)
Two first-class user objects; the internal storage tables are an implementation detail and must never
surface in the UI (no "Built / Saved / Authored / Notebook" wording).
- **Template** = reusable framework with variables. Two internal tables: `prompt_notebooks` (block
  builder, `/build`) and `user_templates` (form editor). Built-in catalog = `data/templates.ts`.
  A Template is always an empty constructor in the fill-in UI. Opening a catalog or user Template
  must not silently restore old answers, select authored defaults, or check default-on options.
- **Prompt** = ready-to-use. Internal: `saved_prompts` (`source_kind ∈ catalog|user|import|manual|ai`;
  standalone rows allowed). A Prompt from a Template renders "**Created from {Template}**" + link
  (`source` on the library item); standalone shows none. A **manual** Prompt stores its markdown in
  `saved_prompts.body` (created/edited via `PromptEditor`; `createManualPromptAction` /
  `updateManualPromptAction`, `bodySchema`). Curated **example** Prompts live in `data/prompts.ts`
  (`EXAMPLE_PROMPTS`, `getExamplePrompt`, `promptCountFor`) — static, SEO-indexed, never collide with a
  catalog slug (enforced by `tests/prompts.test.ts`).
- **Builder creates Templates only** (`/build/template`), but the **create hub** `/build` offers both
  "+ New Template" and "+ New Prompt" (the latter → `/build/prompt` editor). Builder copy says
  "template"; the live preview/health pane stays "prompt" (the *generated* output).
- **Favorites** = bookmarks; targets are `catalog` (Template) or `example_prompt` (Prompt) — a plain
  text `target_kind`, no migration. `lib/bookmarks/{schema,map,actions}.ts` + `BookmarkButton` cover both.
- **My Library** (`app/my/page.tsx`) merges both objects via `lib/library/list.ts`
  (`buildLibrary`/`filterLibrary`, pure + unit-tested) into one card model. The tab bar shows **four**
  `PRIMARY_FILTERS` (All/Templates/Prompts/Favorites). Public items render a per-card **Public**
  visibility chip; private items stay quiet on the card. Legacy status deep links (`drafts`,
  `published`, `shared`) normalize to All.
  The header has **one** primary "+ New" → `/build` (the overview is the Template-vs-Prompt chooser; two
  header CTAs duplicated it). Empty states are **filter-aware + actionable** (per-filter copy + the right
  CTA — never a bare panel). Every type chip carries its object glyph via `lib/library/objectMeta.ts`
  (Template = `list`/indigo, Prompt = `code`/prompt-accent) so the two read apart without the label. Old
  `/my/{saved,templates,notebooks,library}` 308-redirect into `/my?filter=…`.
- **Visibility** (one model, both objects, migration `0016`): `visibility ∈ private|public`;
  `share_slug` is minted lazily on first Public save and **persists**. Private rows are owner-only
  even if they retain a legacy slug. Public rows resolve by URL, appear in community/profile
  listings, and are readable only through security-definer exact-slug RPCs. Making a Prompt public
  freezes its text into `saved_prompts.body` and requires body content plus category.
- **Community & profiles (shipped):** username-based public profiles (`/[username]`), community
  discovery/listing, and remix attribution are live — see "Community, profiles & metrics" below.
  Users submit a template to the community via `/submit-template`.
- **Still deferred:** Import (link + paste — `source_kind='import'`) and AI generation
  (`source_kind='ai'`) — both kinds exist in the enum but have no `lib/` module or route yet.
  `user_templates.is_public` still coexists with `visibility` as a legacy column (the planned `0008`
  consolidation was never written); the app uses `visibility` for sharing. *(Manual Prompt creation + the Markdown
  editor + community listing/profiles are all **shipped**, not deferred.)*

## Accounts (Supabase)
Optional; everything checks `isSupabaseConfigured()` (`lib/supabase/env.ts`) first. With the
`NEXT_PUBLIC_SUPABASE_*` env absent, account UI stays off and the anonymous builder is unaffected.
- **Clients:** `lib/supabase/{client,server,middleware}.ts` (`@supabase/ssr`). Server reads run in
  Server Components via RLS-scoped repos (`lib/*/repo.ts`, `import "server-only"`); **all mutations
  are Server Actions** (`lib/auth/actions.ts`, `lib/userTemplates/actions.ts`,
  `lib/savedPrompts/actions.ts`), Zod-validated. `middleware.ts` refreshes the session + guards
  `/my`, `/account`. Authorize with `getUser()` (revalidates), never `getSession()`. Client chrome
  that only needs display auth state uses `GET /api/account-state` after mount; that route reads the
  server auth cookie and profile row so local dev matches production without calling `cookies()` in
  the root layout.
- **Authz = Row-Level Security** (owner = `auth.uid()`) across the migrations, applied in order.
  There is **no `0008`** (it was skipped — the planned `is_public` cleanup never happened):
  `0001_accounts` (`profiles`, `user_templates`, `saved_prompts`; new-user trigger;
  `delete_current_user` security-definer RPC — so no service-role key is used) · `0002_drop_generated_text`
  (drops the unused `saved_prompts.generated_text` — answers are the truth, prompt text is derived) ·
  `0003_entitlements` (account-bound Pro) · `0004_security_hardening` · `0005_ratings_bookmarks_notebooks`
  (ratings, bookmarks, `prompt_notebooks`) · `0006_notebook_versions_sharing` (version history +
  `share_slug` + `shared_notebook`) · `0007_content_model` (unified Templates/Prompts `visibility` +
  `share_slug` + `shared_*` RPCs) · `0009_interaction_metrics` (usage "Uses" tracking) ·
  `0010_profiles_and_remix` (`profiles.bio`, legacy `profiles.is_public`, `saved_prompts.remixed_from`) ·
  `0011_community_rpcs` (`published_*`/`community_*`/`public_profile*` listing RPCs) ·
  `0012_community_prompt_id` · `0013_profile_content_preview` · `0014_prompt_category_and_listing`
  (`saved_prompts.category` + listing-RPC reshape) · `0015_username_accounts` (required usernames,
  `username_available`, root profile URLs, and username-based public profile RPCs) ·
  `0016_private_public_visibility` (maps `published` to `public`, maps `draft`/`unlisted` to
  `private`, and gates all public RPCs on `visibility = 'public'`). Apply migrations
  *before* deploying matching code, and keep `lib/supabase/types.ts` (the TS schema source of truth)
  in sync with the SQL.
- **Reuse:** a `user_templates` row → catalog `Template` via `lib/userTemplates/map.ts`, so the same
  `Builder`/`buildPrompt` run on a custom template. A `saved_prompts` row → `Answers` via
  `lib/savedPrompts/map.ts` (`rowToAnswers`): it validates the stored JSONB against `answersSchema`
  and **prunes keys for fields/checkboxes no longer on the template** (no stale-key injection). The
  open path (`app/my/prompts/[id]`) recomputes the prompt from answers + the current template.
- Validation is pure + shared by actions and tests (`lib/userTemplates/validate.ts`,
  `lib/savedPrompts/schema.ts`, `lib/auth/schemas.ts`).

## Drafts (anonymous autosave)
The builder autosaves in-progress answers to `localStorage` (`lib/drafts/{draft.ts,useDraft.ts}`,
keyed `easyprompt.draft.<templateId>`) so a logged-out user doesn't lose work on refresh — the
account-free counterpart to `saved_prompts`. SSR-safe (restore *after* mount, no hydration
mismatch), debounced, size-capped, corrupt-safe. Draft restore is **opt-in** for `Builder` surfaces:
catalog/user Template pages start from `blankAnswers` and clear stale local drafts so Templates stay
empty constructors. **Off when reopening a saved prompt** (`initialAnswers` set — that row wins);
cleared on a successful Supabase save (via `onSaved`).

## Pro entitlement (account-bound + anonymous)
Pro/premium is a stateless HMAC token (`lib/access/token.ts`) the client holds in
`localStorage["easyprompt.premium"]`. Two ways to get one, **one consumer** (`/api/premium` +
`lib/premium/client.ts`, both unchanged):
- **Anonymous:** paste a code → `POST /api/entitlement` mints a token → stored with `origin:"code"`.
- **Account-bound (logged in):** `redeemEntitlementAction` (`lib/entitlements/actions.ts`) runs the
  same `getProvider().check()` server-side and upserts an `entitlements` row (RLS owner-only; one per
  user; stores a `code_hash`, never the raw code). `getEntitlementToken()` mints the same token from
  that row; `AuthProvider` calls it on login and adopts it (`origin:"account"`) so Pro **follows the
  user across devices**. Sign-out drops an `account`-origin token; a `code`-origin token survives.
  Manage it on `/account` (`ProSection`). *(This deliberately supersedes the old "Pro stays separate"
  rule — it's now additive: account-bound when logged in, device-local when anonymous.)*

## Community, profiles & metrics
All public read paths are **security-definer RPCs** (`search_path=''`, granted to `anon` +
`authenticated`); RLS on the base tables stays owner-only, so these RPCs are the *only* way anyone
reads another user's rows. Author identity (`username`/`display_name`) is exposed when the author has
a username. Content exposure is still controlled per Template/Prompt by `visibility`.
- **Discovery / listing:** `published_prompts` / `published_templates` (legacy RPC names; public
  rows only, newest-first, paginated;
  short `preview` only, full body fetched lazily on copy) back the community browse grids
  (`lib/community/`, `lib/browse/`). Detail-by-slug via `community_prompt` / `community_template`.
- **Public profiles:** `/[username]` → `public_profile` (+ on-read `reputation`, a 90-day capped
  sum of distinct anonymized actors) and `public_profile_content` (the author's public cards).
  `lib/profiles/repo.ts`. Username is required at signup; legacy `/u/[username]` redirects to
  `/[username]`. The signed-in avatar menu only opens after account state includes a username and
  uses Lucide icons in this order: Profile, My library, Settings, Pro, Sign out. If a legacy loaded
  account has no username, the avatar links to `/account#profile` instead of opening the menu.
- **Remix:** "Use as starting point" forks a community Prompt into the user's library, recording a
  structured `saved_prompts.remixed_from` pointer (the remixer can't read the source row under RLS).
- **Usage metrics ("Uses"):** `POST /api/track` (Node, fire-and-forget, 204) → `record_interaction`
  RPC. "Uses" = copies + open-ins; views are a denominator only. `actor_hash`/`ip_hash` are
  HMAC-salted server-side with `METRICS_SALT` over the session id + a coarsened IP — **no raw PII
  reaches the DB**. `interaction_events` + `content_stats` have RLS on with **zero** grants; only the
  RPCs touch them. Off (accept-and-drop) when `METRICS_SALT` is unset. `lib/metrics/`.
- **Personalization:** `lib/personalization/affinity.ts` ranks catalog items client-side from local
  interaction signals — pure, no server state.
- **Rate limiting:** `lib/rateLimit.ts` (`checkRateLimit`) is a coarse in-memory per-instance throttle
  on all three `/api` routes (`entitlement` 10/min, `premium` 60/min, `track` 120/min). Swap for
  Upstash/KV behind the same signature for a durable cross-instance quota.

## Background system
A single ambient field underlies every route: a faint 8px pixel grid (`body::before`, pure CSS) plus
a soft indigo glow that trails the cursor and lights the grid beneath it
(`components/BackgroundField.tsx` → `.cursor-glow`). Both sit at `--z-field` (−1), below all content.
Full spec in `docs/DESIGN-SYSTEM.md` §11. Working rules:
- The grid is a *design-system field*, not a page effect — it lives in `globals.css`/the layout and
  is consistent across routes. For it to show, page roots must stay `background: transparent`
  (the picker/builder/content roots were switched from `--bg-2` for this). Dense reading/form
  content stays on opaque `.panel`/`.code-well` surfaces so text never sits on the texture.
- Both field layers live in one stacking context below content and are `pointer-events:none`; they
  must never paint over a card, input, or nav. New fixed/elevated chrome gets a `--z-*` token at or
  above 0.
- The glow is desktop-only (`hover:hover and pointer:fine`) and fully off under reduced motion
  (listeners never attach **and** CSS `display:none`). It's a soft indigo radial below content, so it
  lights only the gridded background — it must not tint UI.

## Adding a template
Append an object to `TEMPLATES` in `data/templates.ts`. It must satisfy the invariants the tests
enforce (`tests/templates.test.ts`):
- unique `id` and `slug`
- `category` exists in `CATEGORIES`
- `icon` is a member of the `IconName` union in `components/Icon.tsx`
- field ids and checkbox ids are unique within the template
- `defaultAnswers` must build a non-empty prompt with no leftover `[BRACKET]` placeholders

Then run `npm run test`. Adding a *new* icon means editing BOTH `components/iconNames.ts` (the
runtime `ICON_NAMES` source) and the `VALID_ICONS` mirror in `tests/templates.test.ts`.

## Gotchas (Next.js App Router)
- **All imported CSS is global.** Co-locating a `page.css` per route does NOT scope it; generic
  class names (`.card`, `.layout`) collide across routes. Fix: prefix every rule with a page-root
  class (`.picker-page .card`, `.builder-page .card`). Watch specificity too — a base `.nav .links`
  (0,2,0) beats a mobile override `.nav-links` (0,1,0) even inside a media query; use
  `.nav .nav-links`.
- **Root `not-found.tsx` renders with empty CSS in `next dev`** (unstyled black page). Dev-only
  quirk; `next build`/`next start` prerenders `/_not-found` with the full stylesheet. Verify the 404
  against a production server, not dev.
- **The cursor glow only reads on the grid, and the grid only shows on a transparent root.**
  The glow (`.cursor-glow`) is a soft indigo radial *below* content; if a page paints an opaque
  background over `body::before`, both the grid and the glow vanish there. Keep page roots
  `background: transparent` (see §11) so the field stays consistent across routes.
- **Never read the signed-in user in the root layout.** Calling `getServerUser()` / `cookies()` in
  `app/layout.tsx` forces *every* route dynamic — it silently flips the marketing pages and the SSG
  catalog from static to `ƒ`. Nav auth state is resolved **client-side** (`lib/supabase/useUser.ts`,
  same hydrate-after-mount pattern as `PremiumProvider`); per-request user reads live only in `/my`
  and `/account`. Confirm with the `npm run build` route table (○/● vs ƒ).
- **`NEXT_PUBLIC_*` are inlined at BUILD time, not runtime.** They must be present in the build env
  (e.g. Vercel) or the client half thinks accounts are off even if the server has them. Also: don't
  run `next build` while a dev server is live (drops CSS / corrupts `.next`) — restart dev afterward.

## Testing
Vitest, node environment, `tests/**/*.test.ts` (32 files, 247 tests). Core logic (`lib/`, `data/`) is pure, so
no DOM and no live Supabase is needed — everything tested is a validator, mapper, schema, or pure
helper. Coverage spans prompt assembly (`buildPrompt`), catalog integrity (`templates`, `prompts`),
the Pro code/token scheme (`access`, `providers`, `entitlements`), persistence mappers
(`userTemplates.validate`, `savedPrompts.map`/`.schema`, `drafts`, `notebookDraft`), the block builder
(`blocks.*`), the content/library layer (`library.list`, `objectMeta`, `publishGuard` public visibility guard,
`notebooks.share`), community/metrics (`community.map`, `browse.map`, `metrics.*`, `ratings.*`,
`bookmarks`, `personalization.affinity`, `profile.schema`), `markdown.highlight`, `rateLimit`, and
`auth.validation`. **Gaps:** no API-route handler tests and no e2e/a11y suite yet.
